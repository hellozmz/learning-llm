第8课：Function Calling / Tool Use

模型不是只能"说话"——它可以选择调用什么函数、传什么参数。但注意：执行代码的是你，不是模型。
预计时间：40 分钟  |  2026-06-26

1. 最关键的澄清：模型并不"调用"函数

"Function Calling"这个名字有误导性。模型没有执行任何代码——它做的事情非常朴素：

Function Calling 的真相: 用户问: "沪深300今天涨了多少？" ↓ LLM 输出: stop_reason: "tool_use" { "name": "get_stock_price", "arguments": { "index": "沪深300", "field": "change_pct" } } ↓ 你的代码 解析这个 JSON → 调用真正的 get_stock_price() → 拿到结果 ↓ 把结果拼回消息列表 → 再次请求 LLM ↓ LLM 输出: "沪深300今日上涨0.82%。"
核心认知：Function Calling 本质上是"模型输出了特定格式的 JSON，而不是自然语言"。你写的代码是执行者、决策者和安全边界。模型只是建议"我可能需要这个工具、用这些参数"，最终是否执行、怎么执行，完全由你的应用层控制。

2. 工具定义：你告诉模型"能做什么"

模型不是天生知道你有什么工具。你需要用 JSON Schema 格式描述每个工具：

# 一个可转债数据查询工具的定义（OpenAI 兼容格式）
tools = [
  {
    "type": "function",
    "function": {
      "name": "query_cb_data",
      "description": "查询可转债的行情数据和条款信息。输入转债代码或名称，返回价格/溢价率/条款触发状态。",
      "parameters": {
        "type": "object",
        "properties": {
          "cb_name": {
            "type": "string",
            "description": "可转债名称或代码，如'浦发转债'或'110059'"
          },
          "data_type": {
            "type": "string",
            "enum": ["price", "premium_rate", "clause_status", "all"],
            "description": "要查询的数据类型"
          }
        },
        "required": ["cb_name"]
      }
    }
  }
]

模型怎么"学会"使用这个工具？不是微调，也不是训练——就是靠 System Prompt 里的工具描述 + 指令微调阶段的 tool-use 训练。模型在预训练时见过大量 API 调用格式的文本，推理时你把工具定义放进请求里，它就能产出符合格式的 JSON。

3. 工具描述的工程要点

写好工具描述和写 prompt 一样，有坑：

原则坏的描述好的描述
说清楚"什么时候用" "查询数据" "当用户问到可转债的实时价格、溢价率或条款触发状态时使用"
参数名要直观 param1: string cb_name: string — 可转债名称或代码
enum 限制可选值 "type": "string" "enum": ["price","premium_rate","clause_status"]
给出参数示例 无 "如 cb_name='浦发转债', data_type='price'"
明确约束边界 无 "仅用于查询已有数据，不执行任何修改操作"
安全红线：永远不要把 execute_sql(raw_query) 这样的工具暴露给模型。模型不知道你的表结构（它猜的）、不知道索引（它不管）、不知道你到底有没有 DELETE 权限。正确做法是把每个查询封装成独立的、参数类型明确的具体函数——"数据库查询"不是一个工具，是五十个工具。

4. 完整对话循环

一次 Function Calling 的完整消息序列：

# Round 1: 用户提问 → 模型决定调用工具
messages = [
  {"role": "system", "content": "你是可转债分析助手..."},
  {"role": "user",   "content": "浦发转债现在溢价率多少？接近强赎触发吗？"}
]

response = client.chat.completions.create(
  model="gpt-4o", messages=messages, tools=tools
)
# response.choices[0].finish_reason == "tool_calls"
# response.choices[0].message.tool_calls = [{
#   "function": {"name": "query_cb_data",
#                "arguments": '{"cb_name":"浦发转债","data_type":"all"}'}
# }]

# Round 2: 你的代码执行函数，结果追加到消息列表
tool_result = query_cb_data("浦发转债", "all")
# → {"price": 108.5, "premium_rate": 0.23, "clause": {"强赎": "未触发"}}

messages.append(response.choices[0].message)  # 模型的 tool_call 消息
messages.append({
  "role": "tool",
  "tool_call_id": response.choices[0].message.tool_calls[0].id,
  "content": json.dumps(tool_result)
})

# Round 3: 模型拿到结果，生成最终回答
response = client.chat.completions.create(
  model="gpt-4o", messages=messages
)
# → "浦发转债当前价格108.5元，溢价率23%。目前未触发强赎条款..."
关键细节：Round 2 中 model 的 tool_call 消息和 role="tool" 的结果消息都必须追加。这和在 prompt 里手工拼"检索到的文档是..."完全不同——模型在训练时就被教过"tool_calls → tool 结果 → 最终回答"这个消息格式，它知道怎么用。

5. 多工具场景：模型怎么选

当你提供了 10 个工具，模型需要决定：用哪个（或哪几个）、什么顺序、传什么参数。

用户问: "比较一下浦发转债和中信转债的溢价率，哪个更低？" 模型可能的决策路径: 1. 识别需要"溢价率"数据 → 选 query_cb_data 2. 识别需要两个转债的数据 → 并行调用两次 query_cb_data (不同 cb_name，相同 data_type="premium_rate") 或者: 1. 识别"溢价率"是计算型问题 → 选 get_premium_rate("浦发转债") 2. 再选 get_premium_rate("中信转债") 3. 如果结果已经有了，直接生成答案

模型做这个决策靠的是：每层 Transformer 对工具定义文本的注意力。如果你在工具描述里写了"用于查询个别转债的溢价率"，模型就不会尝试用它查"所有转债"。如果你写了"支持批量查询"，模型可能会一次传多个参数。

工具描述就是模型"理解工具"的唯一入口。没有别的。

6. 并行工具调用

多个互不依赖的工具调用可以并行发出——这是降低延迟的关键设计：

# 模型可以一次返回多个 tool_calls
response.choices[0].message.tool_calls = [
  {"function": {"name": "query_cb_data", "arguments": '{"cb_name":"浦发转债","data_type":"price"}'}},
  {"function": {"name": "query_cb_data", "arguments": '{"cb_name":"中信转债","data_type":"price"}'}},
  {"function": {"name": "get_market_index", "arguments": '{"index":"中证转债"}'}}
]

# 三个查询不互相依赖 → 可以并发执行
import concurrent.futures
with concurrent.futures.ThreadPoolExecutor() as executor:
    results = list(executor.map(execute_tool, tool_calls))
场景串行延迟并行延迟
3 个独立 API 查询（每个 200ms）600ms200ms
先查条款 → 再根据条款结果决定下一步400ms（必须串行）无法并行
工程判断：不是所有调用都能并行。模型会区分"需要等上一个结果"和"独立查询"。如果模型连续两次请求都做了 tool_calls，说明第一轮的结果影响了第二轮的选择——这是依赖关系，不能强行并行化。

7. 自检

问题 1/4 — 概念辨析

下面关于 Function Calling 的描述，哪一个是正确的？

A — 模型在推理时会实际调用你注册的函数并获取执行结果
B — 模型通过微调学会了执行你代码中定义的 Python 函数
C — 模型读取函数源码后在其内部沙箱中模拟运行结果
D — 模型生成一个结构化 JSON 输出，应用代码解析后决定是否执行

问题 2/4 — 工具设计判断

你为可转债系统定义了两个工具。用户问"哪些银行转债的溢价率低于15%"，模型只调用了 query_cb_list 但没有调用 calculate_premium，最终给出了错误的答案。以下哪项最可能是根因？

A — 模型的工具选择算法有 bug，需升级到更新的模型版本
B — 系统提供的工具数量过多导致模型选择困难而漏调
C — calculate_premium 的描述让模型误判它不适用于筛选举例场景
D — Function Calling 模式下模型只能选择唯一的一个工具执行

问题 3/4 — 循环判断

考虑以下消息序列。哪一轮消息组合是 Function Calling 中必须按顺序出现的？（role 标记：user / assistant(tool_calls) / tool / assistant）

A — user → tool → assistant → assistant 三个角色即可完成闭环
B — user → assistant(tool_calls) → tool → assistant 缺一不可
C — user → tool → user → assistant 两次 user 保证参数正确
D — system → tool → assistant → user 从 system 直接跳工具调用

问题 4/4 — 安全设计

以下哪个工具定义最可能造成安全问题，不应该暴露给 LLM？

A — 根据转债名称返回该转债的实时价格和溢价率等行情数据
B — 根据日期范围返回该时段内的转债市场涨跌统计汇总
C — 接收 SQL 查询语句并直接在转债数据库中执行以返回任意结果
D — 根据用户指定的筛选条件从数据库检索匹配的转债列表
