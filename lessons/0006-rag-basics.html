第6课：RAG 基础 —— 检索增强生成全链路

Embedding → 向量数据库 → 检索 → 拼进 Prompt → 生成，五步拆解
预计时间：35 分钟  |  2026-06-22

1. 为什么模型不能直接读文档？

两个根本原因：

1. 上下文窗口有上限。 即使 128K 窗口的模型，一份 200 页的转债募集说明书（约 400K tokens）也塞不进去。

2. 即使塞进去了，模型也会"迷失"。 研究证明，当上下文中有大量不相关信息时，模型准确率显著下降——这叫做"Lost in the Middle"问题。把整本说明书塞进去，模型反而找不到关键条款。

RAG 解决的就是这个矛盾：先检索出相关的部分，再让模型基于这些精选内容生成回答。

2. RAG 完整五步流程

┌─────────────────────────────────────────────────────────────┐ │ 1. 文档预处理（离线） │ │ PDF/网页 → 解析 → 切块(Chunking) → Embedding → 存入向量库 │ ├─────────────────────────────────────────────────────────────┤ │ 2. 用户提问（在线） │ │ "这只转债的下修条款触发条件是什么？" │ │ ↓ │ │ 用户问题 → Embedding → 查询向量 │ ├─────────────────────────────────────────────────────────────┤ │ 3. 向量检索 │ │ 查询向量 → 向量数据库 → Top-K 最相似的 Chunk │ │ → [条款文档_第27段, 条款文档_第35段, 公告_2025-03-01] │ ├─────────────────────────────────────────────────────────────┤ │ 4. 拼装 Prompt │ │ System Prompt + 检索到的 Chunks + 用户问题 → 完整 Prompt │ ├─────────────────────────────────────────────────────────────┤ │ 5. 生成回答 │ │ 完整 Prompt → LLM → "该转债的下修条款规定：在任意连续30个..." │ └─────────────────────────────────────────────────────────────┘

3. 核心组件一：Embedding 模型

Embedding 模型把一段文字变成一个固定长度的向量。语义相近的文字，向量也相近。

# 伪代码
text1 = "可转债下修条款触发条件"
text2 = "正股价格连续30个交易日低于转股价的70%"
text3 = "今天天气很好"

emb1 = embedding_model.encode(text1)  # → (1024,) 的向量
emb2 = embedding_model.encode(text2)  # → (1024,) 的向量
emb3 = embedding_model.encode(text3)  # → (1024,) 的向量

cosine_sim(emb1, emb2)  # 0.85  ← 语义相关
cosine_sim(emb1, emb3)  # 0.12  ← 语义无关
模型维度适用场景选择理由
BGE-M3 (BAAI)1024 中文通用，多语言 中文 RAG 首选，支持稠密+稀疏混合检索
text2vec-large-chinese1024 中文语义匹配 轻量，适合本地部署
OpenAI text-embedding-3256~3072 多语言，高精度 API 调用，贵但省心
Jina Embeddings v31024 长文档，多语言 支持 8192 token 输入，适合长 chunk
选型原则：中文场景→BGE-M3 是事实标准。精度要求高→OpenAI。本地部署→text2vec。长文档→Jina。不要同时用多个——一个够好就行，工程复杂度 > 那 2-3 个点的精度提升。

4. 核心组件二：Chunking（文档切分）

这是 RAG 里最被低估但最重要的步骤。切得太小丢失上下文，切得太大检索精度下降。

策略大小适用场景
固定大小切分 512 tokens 通用场景，实现简单
语义切分 按段落/章节 文档有清晰结构（财报、说明书）
递归切分 先按章节→再按段落→再按句子 需要多层级检索
父子切分 (Parent-Child) 小块检索(128t) + 大块返回(1024t) 精度和上下文的折中方案 
# Chunking 的关键参数
chunk_size = 512        # 每个 chunk 的 token 数
chunk_overlap = 64      # 相邻 chunk 的重叠量

# 为什么需要 overlap？
# Chunk A: "...当正股价格在任意连续30个交"
# Chunk B: "30个交易日中至少有15个交易日..."
# ↑ 没有 overlap，"连续30个交易日"这个关键短语被切断了
# 有了 overlap：
# Chunk A: "...当正股价格在任意连续30个交易日"
# Chunk B: "任意连续30个交易日中至少有15个交易日..."

5. 核心组件三：向量数据库

向量数据库做的事：存 N 个向量 → 给定一个查询向量 → 用近似最近邻算法(ANN)快速找到最相似的 K 个。

数据库类型适用场景
ChromaDB嵌入式，Python 原生开发/原型，数据量 < 100K 条
Milvus分布式，高性能生产环境，亿级向量
Qdrant高性能，Rust 编写需要高并发低延迟
FAISS (Meta)纯向量索引库嵌入已有系统，不提供持久化
pgvectorPostgreSQL 插件已有 PG 基础设施，简单 RAG
Elasticsearch全文检索 + 向量需要 BM25 + 向量的混合检索
选型建议：你的可转债项目数据量可能几千到几万份文档 → ChromaDB 足够了。如果后续要做全市场实时监控 → 上 Milvus。关键是不要过早优化数据库——先跑通链路，再换数据库。

6. 核心组件四：检索策略

拿到向量之后，不是找最相似的 Top-K 就完了。工程上有三个关键选择：

# 1. 相似度计算方式
余弦相似度 (Cosine): 最常用，只关心方向不关心长度
点积 (Dot Product): 需要 Embedding 已归一化
欧氏距离 (Euclidean): 对向量绝对值敏感

# 2. Top-K 的选择
K=3:  精确但可能遗漏相关信息
K=10: 覆盖率好但噪声增多，给模型增加了"阅读理解"负担
K=5:  工程上的常用折中

# 3. 检索后处理
去重: 相似度 > 0.95 的 chunk 只保留一个
排序: 可以引入 Reranker 做二次精排（第7课讲）
上下文扩展: 检索 chunk 的相邻 chunk 也一起返回

7. 最小可运行 RAG 示例

from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.embeddings import HuggingFaceEmbeddings
from langchain_community.vectorstores import Chroma

# 1. 加载文档
with open("可转债下修条款.txt") as f:
    documents = [f.read()]

# 2. 切分
splitter = RecursiveCharacterTextSplitter(
    chunk_size=512, chunk_overlap=64
)
chunks = splitter.create_documents(documents)

# 3. Embedding + 向量库
embedding = HuggingFaceEmbeddings(
    model_name="BAAI/bge-m3"
)
vectorstore = Chroma.from_documents(
    chunks, embedding, persist_directory="./cb_db"
)

# 4. 检索
query = "什么条件下触发下修？"
docs = vectorstore.similarity_search(query, k=5)

# 5. 拼 prompt + 调 LLM
context = "\n\n".join([d.page_content for d in docs])
prompt = f"基于以下文档内容回答问题：\n{context}\n\n问题：{query}"
# response = llm.invoke(prompt)
这就是一个能跑的最小 RAG 系统——不到 10 行核心代码。实际生产环境还需要：更好的 chunking 策略、Reranker、检索质量评估、缓存。但骨架就是这个。

8. 自检

问题 1/4

RAG 解决的核心矛盾是什么？

A. 模型推理太慢
B. 上下文窗口有限 + 无关信息会降低模型准确率（Lost in the Middle）——需要先检索再生成
C. 模型不认识中文词汇
D. 向量数据库比 SQL 数据库快

问题 2/4

Chunk Overlap（重叠）的作用是什么？

A. 让 chunk 数量更多
B. 防止关键语义单元被切在两个 chunk 的边界上——用少量冗余换语义完整性
C. 提高向量检索速度
D. 减少 Embedding 计算量

问题 3/4

Top-K=50 而不仅仅是 K=5 会有什么负面影响？

A. 向量检索会报错
B. 大量低相关度 chunk 进入上下文，既浪费 token 预算，又增加噪声——模型在几十个碎片中挑不出真正关键的信息
C. Embedding 模型会过载
D. 向量数据库无法返回超过 10 条结果

问题 4/4

你的可转债项目有 5000 份募集说明书和 8000 份公告，ChromaDB 够用吗？什么情况下需要换 Milvus？

A. 不够，必须用 Milvus
B. 够用——ChromaDB 轻松处理 10 万条级数据。需要换 Milvus 的信号是：单次查询 > 100ms、数据量 > 100 万条、需要分布式部署和实时插入
C. ChromaDB 不支持中文
D. 两个数据库功能完全一样，随便选
