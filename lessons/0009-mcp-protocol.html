第9课：MCP —— 让工具接入有标准可循

你会写 Function Calling 了。但如果每个 LLM 应用都要重新写一遍工具集成代码，你会发疯的。
预计时间：35 分钟  |  2026-06-26

1. 问题：Function Calling 的"最后一公里"

上节课你理解了 Function Calling：Python 脚本定义工具 → LLM 输出 JSON → 你的代码执行 → 结果回送。但有个工程问题：

场景：你写了 3 个可转债数据查询工具 App A (Claude Desktop) App B (自研量化平台) App C (LangChain Agent) │ │ │ ▼ ▼ ▼ 要集成你的工具 要集成你的工具 要集成你的工具 看不懂你的代码 看不懂你的代码 看不懂你的代码 │ │ │ ▼ ▼ ▼ 你重写一遍集成 你又重写一遍 你再重写一遍

每次换一个 LLM 客户端，你都要重新写"怎么把你的工具注册进去、怎么处理调用、怎么返回结果"这套胶水代码。

MCP 解决的就是这个问题：定义一套标准协议，让你写一次工具服务端，任何兼容 MCP 的客户端都能直接用。

2. MCP 架构：Client — Server

MCP 的物理布局: ┌─────────────────────┐ ┌──────────────────────┐ │ MCP Client │ │ MCP Server │ │ (Claude Desktop, │ stdio │ (你写的 Python 脚本) │ │ Cursor, 自研 App) │◄──────►│ │ │ │ JSON │ 暴露: │ │ - 管理连接 │ RPC │ • tools: 函数列表 │ │ - 发现工具 │ │ • resources: 数据源 │ │ - 发起调用 │ │ • prompts: 提示词模板 │ └─────────────────────┘ └──────────────────────┘

核心是三个原语（primitives）：

原语做什么在可转债场景的例子
tools 可调用的函数，LLM 可以选择执行 query_cb_data, calculate_premium
resources 暴露数据给客户端读取（类似 GET 请求） cb_docs://募集说明书/浦发转债.pdf
prompts 预定义的提示词模板 "分析这只转债的条款风险"模板
和 Function Calling 的关系：MCP 的 tools 原语 = Function Calling 的工具定义，但用标准协议暴露。Client 启动时发一个 tools/list 请求 → Server 返回工具列表 → Client 把这些工具注册给 LLM → LLM 做 Function Calling 决策 → Client 通过 MCP 协议把调用请求发给 Server 执行。MCP 是传输层，Function Calling 是决策层。

3. 一个可运行的 MCP Server

# cb_server.py — 可转债数据 MCP Server
from mcp.server import Server, stdio_server
from mcp.types import Tool, TextContent

server = Server("cb-data-server")

# 定义工具
@server.list_tools()
async def list_tools() -> list[Tool]:
    return [
        Tool(
            name="query_cb_price",
            description="查询指定可转债的当前价格和溢价率",
            inputSchema={
                "type": "object",
                "properties": {
                    "cb_name": {
                        "type": "string",
                        "description": "可转债名称或代码"
                    }
                },
                "required": ["cb_name"]
            }
        )
    ]

# 实现工具调用
@server.call_tool()
async def call_tool(name: str, arguments: dict) -> list[TextContent]:
    if name == "query_cb_price":
        price_data = your_query_function(arguments["cb_name"])
        return [TextContent(
            type="text",
            text=json.dumps(price_data, ensure_ascii=False)
        )]

# 启动
async def main():
    async with stdio_server() as (read, write):
        await server.run(read, write)

asyncio.run(main())

关键点：

  • @server.list_tools() —— 告诉客户端"我有哪些工具"，等价于 Function Calling 里的 tools 数组定义
  • @server.call_tool() —— 客户端通过 MCP 协议发过来的调用请求，等价于你写的 if tool_name == "xxx" 分支
  • stdio_server() —— 通过标准输入/输出通信，客户端启动你的脚本作为子进程

4. 传输层：stdio vs SSE

stdio (标准输入输出)SSE (Server-Sent Events)
通信方式父进程 ↔ 子进程，stdin/stdout 传 JSONHTTP + 长连接推送，客户端连服务端 URL
适用场景本地工具，个人使用团队共享，远程服务
部署方式客户端启动时自动拉起 Python 进程作为常驻 HTTP 服务运行在某台机器上
启动复杂度零（客户端负责启动）需要部署服务并暴露端口
实践建议：个人开发用 stdio（零部署），团队共享换 SSE。同一个 Server 代码两种传输都支持，只是启动方式不同。

5. MCP 和 Function Calling 的协作流程

当用户在 Claude Desktop 里问"浦发转债现在多少钱？"，完整的调用链是这样的：

1. 客户端启动时: Client → tools/list → Server → 返回工具列表 2. 用户提问后: Client 把工具列表 + 用户问题 → 发给 LLM 3. LLM 返回: {"tool_calls": [{"name": "query_cb_price", "arguments": {"cb_name":"浦发转债"}}]} 4. Client 通过 MCP: Client → tools/call → Server → 执行 query_cb_price() → 返回 {"price": 108.5} 5. Client 把结果发回 LLM → LLM 生成: "浦发转债当前 108.5 元。" 关键: 第 3 步是 Function Calling，第 4 步是 MCP 协议。 MCP 不替代 Function Calling——它规范围绕 Function Calling 的"前后步骤"。

6. 为什么 MCP 是 2026 的标准

不因为它是 Anthropic 提出的。因为一个简单事实：

类比：USB 协议出现之前，每个外设都有自己的接口。USB 之后，一个标准接口搞定键盘鼠标U盘。MCP 就是 AI 工具接入的 USB——你写一次 Server，任何 Client 都能用。没有 MCP 之前，给每个 LLM 客户端适配工具 = 给每个电脑配不同的键盘接口。

2026 年数据：MCP 月下载量 97M，已注册 MCP Server 超过 2000 个。Claude Desktop、Cursor、Continue、Sourcegraph Cody 等主流 AI 工具都内置了 MCP Client。不是你需不需要学 MCP——是你不学就只能在自己写的 Python 脚本里玩 Function Calling，没法接入整个生态。

7. 自检

问题 1/4 — 定位辨析

MCP 协议和 Function Calling 的关系，以下哪项描述最准确？

A — MCP 是 Function Calling 的替代方案，两者选一个用即可不必并存
B — MCP 使用自己的模型推理机制完全取代了 Function Calling 流程
C — MCP 标准化工具的发现与调用传输，Function Calling 负责模型侧决策
D — MCP 只负责提示词模板管理，Function Calling 负责工具的参数填充

问题 2/4 — 流程排序

用户在 MCP 客户端中提问，以下事件发生的正确时间顺序是：① LLM 决策调哪个工具 ② Client 通过 MCP 协议向 Server 请求工具列表 ③ Server 执行函数返回结果 ④ Client 把工具定义注册进 LLM 请求

A — ①→②→③→④ LLM 决策最先触发随后才去获取和注册工具
B — ②→④→①→③ 先发现工具再注册给 LLM，LLM 决策后 Server 执行
C — ④→①→②→③ 先注册工具 LLM 决策后才发现 MCP Server 去调用
D — ②→③→①→④ 发现工具后立刻执行函数然后 LLM 再决定用哪个

问题 3/4 — 工程判断

你写了一个可转债数据 MCP Server。自己测试时用 stdio 传输，现在要分享给团队用。以下哪项是换到 SSE 传输的关键原因？

A — stdio 模式下工具定义的 JSON Schema 校验不够严格易出错
B — SSE 传输速度比 stdio 快 10 倍以上适合生产环境使用
C — stdio 模式不支持 tools、resources、prompts 三个原语
D — stdio 要求 Server 作为客户端子进程启动，团队共享需独立部署的服务

问题 4/4 — 设计判断

同事说"我们不需要 MCP，直接用 Function Calling 就够了"。以下哪种场景下他的话不成立？

A — 只在一个 Python 脚本中使用 LLM 调用自己定义的几个本地工具
B — 工具逻辑极其简单仅涉及字符串处理无需外部数据查询
C — 需要让同一套工具被多个不同的 LLM 客户端发现和使用而避免重复集成
D — 所有工具调用都是串行依赖关系无法利用并行优化减少延迟
