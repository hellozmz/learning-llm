第5课：从 Prompt Engineering 到 Context Engineering

2026 年的范式转移：写好提示词不重要，设计好上下文才重要
预计时间：30 分钟  |  2026-06-22

1. 为什么"写提示词"正在过时？

2023-2024 年，大家都在研究怎么把一句话写得更好让 GPT 产出更好的结果。但 2026 年的共识是：单条 prompt 的质量天花板很低——真正决定效果的是模型在推理时能"看到"什么信息。

Prompt Engineering (2023): "你是一个金融分析师。请分析这只转债的..." → 全靠一句话的质量 Context Engineering (2026): 系统自动检索了: ✓ 这只转债的条款文档（下修/强赎/回售） ✓ 正股最近 30 天走势 ✓ 同行业转债的溢价率对比 ✓ 上次触发强赎时的公告内容 → 然后把这些 + 用户问题一起发给模型 → 模型看到的是"完整上下文"，不是"一句聪明的指令"
核心转变：Prompt Engineering 是在教模型"怎么想"。Context Engineering 是给模型"足够的信息让它自己想"。前者靠语言技巧，后者靠系统工程。

2. Prompt Engineering 仍然有用的部分

不是说 prompt 不重要了，而是它从"全部"退到了"基础设施"的角色。以下技术仍然有效且常用：

System Prompt（角色设定）

# 好的 System Prompt 特征：
# 1. 定义角色边界
# 2. 约束输出格式
# 3. 声明能力范围

你是一个可转债投资分析助手。你的职责：
- 根据提供的条款文档和行情数据进行分析
- 当数据不足时，明确告知用户"需要补充XX信息"
- 输出使用 Markdown 格式，关键数字加粗
- 不提供买卖建议，只做客观分析

Few-shot（示例驱动）

# 给模型看几个例子，比描述规则更有效
输入: 转债A，价格110，溢价率15%，正股涨2%
输出: 转债跟涨，涨幅0.8%，偏股性

输入: 转债B，价格95，溢价率80%，正股涨2%
输出: 转债几乎不动，涨幅0.1%，偏债性，高溢价压制

输入: 转债C，价格108，溢价率20%，正股跌3%
输出: ?

Chain-of-Thought（思维链）

# 让模型"一步步想"，显著提高推理准确率
"请一步步分析这只转债是否值得关注：
 1. 先看价格和溢价率
 2. 再看正股基本面
 3. 然后看条款（下修/强赎触发情况）
 4. 最后综合判断"

# 2026 年新发现：隐式 CoT（Implicit CoT）
# 模型内部已经会推理，不需要你显式写"让我们一步步思考"
# 节省 58% 的生成 token（复旦大学 2026 研究）

3. Context Engineering 的四个维度

维度Prompt Engineering 做法Context Engineering 做法
知识 把知识写在 prompt 里 RAG 系统自动检索相关文档
工具 在 prompt 里描述工具用法 Function Calling + MCP 协议自动接入
记忆 把历史对话拼进 prompt 三层记忆：上下文缓存 → 向量检索 → 持久化记忆
约束 在 prompt 里写规则 结构化输出（JSON Schema）+ Guardrails 层
你的可转债场景举例：用户问"哪些转债的溢价率最近跌得厉害"，不是把"怎么算溢价率"写在 prompt 里，而是设计上下文系统：自动查数据库拿到所有转债的溢价率序列 → 算变化 → 排序 → 把 Top 10 的结果 + 原始数据一起发给模型 → 模型只需做"理解和呈现"。

4. Token 预算管理

Context Engineering 的核心工程约束：模型的上下文窗口有限，每多放一个 token 都花钱。

典型的 Context 预算分配（128K 窗口的模型）: System Prompt: ~500 tokens (角色 + 输出格式) Few-shot Examples: ~2,000 tokens (3-5个高质量示例) 检索到的文档: ~80,000 tokens (RAG 召回的核心内容) 对话历史: ~20,000 tokens (最近几轮对话) 用户当前问题: ~500 tokens 留白（安全余量）: ~25,000 tokens (防止截断) ───────────────────────────────────────── 总计: ~128K

原则：System Prompt 保持尽可能短。示例精而不多。检索到的文档按相关度排序，低相关度的截断。历史对话定期压缩（用模型自己摘要）。

问题 1/4

Context Engineering 和 Prompt Engineering 的核心区别是什么？

A. Context Engineering 用更长的 prompt
B. Prompt Engineering 靠写好指令，Context Engineering 靠设计好模型能"看到"的信息系统
C. Context Engineering 不需要 System Prompt
D. Prompt Engineering 只适用于 GPT-3，Context Engineering 适用于 GPT-4

问题 2/4

System Prompt 和 RAG 检索结果同时在上下文里时，System Prompt 应该怎么设计？

A. 越详细越好，把所有规则都写进去
B. 保持精简——定义角色和输出格式即可，具体知识由 RAG 提供，避免浪费 token 预算
C. 只写 System Prompt，不需要 RAG
D. 把 RAG 检索结果直接写进 System Prompt

问题 3/4

为什么在可转债分析场景中，Few-shot 示例特别重要？

A. 因为模型不认识"可转债"这个词
B. 因为金融分析的输出格式和推理路径有行业惯例，给 2-3 个"好的范例"比写 20 条规则更有效
C. 因为 Few-shot 可以替代所有 RAG 检索
D. 因为模型在金融领域没有预训练

问题 4/4

在 128K 上下文中，为什么要在检索文档区域后预留 ~25K 的安全余量？

A. 让模型有"思考空间"
B. 防止模型输出被截断——模型的生成长度也要消耗上下文窗口，不留余量会导致回答被腰斩。同时给后续对话轮次留空间
C. 这是 API 的硬性要求
D. 安全余量不是必需的，可以用满 128K
