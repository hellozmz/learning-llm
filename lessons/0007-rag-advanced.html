第7课：RAG 进阶 —— 从"能跑"到"能打"

混合检索 + Reranker + GraphRAG：纯向量检索在金融文档上召回率只有 50%，剩下的 50% 在哪里
预计时间：40 分钟  |  2026-06-24

1. 你已有的知识：哪些可以复用

如果你做过搜索引擎或 ANN 向量召回，以下概念你已经有肌肉记忆，我们直接带过：

你已知的（搜索引擎）RAG 中的对应物差异
倒排索引 + BM25稀疏检索 (Sparse Retrieval)相同。 BM25 就是那个 BM25，没变
ANN 向量召回 (Faiss/HNSW)稠密检索 (Dense Retrieval)相同。 Embedding → ANN 索引 → Top-K
Learning to Rank (LTR)Reranker (Cross-Encoder)思路相似，实现不同。 LTR 靠特征工程，Reranker 靠 Transformer 做 query-doc 联合编码
知识图谱GraphRAG最大差异。 搜索引擎的 KG 是手工构建的，GraphRAG 用 LLM 自动提取实体和关系
这一课的核心问题：搜索引擎要的是"Top-10 蓝链接"，RAG 要的是"给 LLM 的最优上下文片段"。两者的召回可以复用，但精排和上下文组装是 RAG 独有的工程挑战。

2. 混合检索：BM25 + Dense 的正确配方

为什么纯向量检索不够

一个具体的金融文档场景：

查询: "正股价格连续30个交易日低于转股价70%触发下修" 段落A (相关): "正股价格在任意连续30个交易日中低于转股价的70%时，董事会有权..." 段落B (不相关): "连续20个交易日中至少15个低于转股价的85%时触发回售..." → BGE-M3 embedding 下，两段的余弦相似度 > 0.8 原因: Embedding 模型擅长语义主题匹配，但对精确数值阈值 (30 vs 20, 70% vs 85%) 和法律术语(下修 vs 回售)的区分力不足。

这不是 Embedding 模型的问题——这是稠密向量的固有特性：n 维空间中的向量必然丢失精确词面匹配信息。

BM25 补什么

BM25 不关心语义，只关心词频和稀有度。它不会把"下修"和"回售"搞混，因为它们是完全不同的词。同时它对数值敏感——"30"和"20"是完全不同的 token。

✅ Dense 擅长

"正股大幅下跌触发条款" → 找回"股价走弱，满足下修条件"
语义相近，但措辞完全不同

❌ Dense 薄弱

"连续30个交易日" vs "连续20个交易日"
主题相同(都是"连续交易日条件")，但数值不同——Embedding 分不清

✅ BM25 擅长

"转股价的70%" → 精确匹配 "70%"
稀有 token "70%" 在 BM25 中获得极高权重

❌ BM25 薄弱

"怎么判断这只转债有投资价值" → 找不到"该可转债是否值得买入"
词面完全不同，但语义相同
结论：金融文档 RAG 中，纯向量的召回率通常只有 50-60%，加上 BM25 可以拉到 75-80%（来源：FinBERT-RAG benchmark, 2024）。这不是"加个 BM25 也好"的锦上添花——是必须的。

结果融合：RRF (Reciprocal Rank Fusion)

BM25 返回一个排序列表，Dense 返回另一个排序列表。怎么合成一个？

做法是 RRF，公式很有名但大多数人只看不用：

RRF 公式: score(d) = Σ_{r ∈ retrievers} 1 / (k + rank_r(d)) 其中: - d 是候选文档 - r 是每个检索器（BM25、Dense 可以是两个 r，也可以更多） - rank_r(d) 是文档 d 在检索器 r 中的排名（从 1 开始） - k 是平滑常数，默认 60 直观理解: - k 很大(如 60) → 1/(60+1) ≈ 1/(60+100) 差别不大 → 排名 1 和排名 100 几乎等价 → 更"民主" - k 很小(如 1) → 1/(1+1) = 0.5, 1/(1+100) ≈ 0.01 → 排名靠前的文档压倒性优势 - k=60 是经验值，来源于搜索社区多年实验 
# RRF 实现（核心逻辑就这几行）
def reciprocal_rank_fusion(results_list, k=60):
    """
    results_list: [bm25_results, dense_results, ...]
    每个 results 是 [(doc_id, score), ...] 按分数降序
    """
    scores = {}
    for results in results_list:
        for rank, (doc_id, _) in enumerate(results, start=1):
            scores[doc_id] = scores.get(doc_id, 0) + 1.0 / (k + rank)
    # 按 RRF 分数降序返回
    return sorted(scores.items(), key=lambda x: x[1], reverse=True)
工程陷阱：BM25 的分数是绝对值（和文档长度、词频相关），Dense 的余弦相似度是 [-1, 1]。直接加权求和（alpha * BM25 + beta * cos）需要先做分数归一化，否则某个来源的分数会碾压另一个。RRF 绕过了这个归一化问题——它只看排名，不看绝对分数。

3. Reranker：用 Transformer 做精排

Bi-Encoder vs Cross-Encoder

这是理解 Reranker 的关键。第一阶段检索用的是 Bi-Encoder，第二阶段精排用的是 Cross-Encoder。差别很大：

Bi-Encoder（第一阶段，快速召回）: Query → Encoder → q_vec (1024,) Doc1 → Encoder → d1_vec (1024,) ─┐ Doc2 → Encoder → d2_vec (1024,) ─┤ 独立编码，cos(q, d_i) 计算相似度 DocN → Encoder → dn_vec (1024,) ─┘ 关键：Query 和 Doc 各自独立编码，没有交互。 速度：可以预先算好所有 Doc 向量，在线只算 Query 向量 → ~1ms Cross-Encoder（第二阶段，精准重排）: [CLS] Query [SEP] Doc [SEP] → Encoder → 相关性分数 关键：Query 和 Doc 拼接后一起过 Transformer， 每一层 Attention 都在 Query 和 Doc 的 token 间充分交互。 速度：每对 query-doc 都要过一遍完整 Transformer → ~30ms/对
Bi-EncoderCross-Encoder
Query-Doc 交互 只在最后的点积 每一层都有完整交叉注意力
Doc 向量预计算 可以 不可以（每次 query+doc 拼接都不同）
单次延迟 ~1ms ~30-50ms
可处理候选数 百万级 通常 20-50 个（延迟约束）
代表模型 BGE-M3, text2vec BGE-Reranker-v2, Cohere Rerank v3

两阶段流水线的延迟计算

假设你的系统从 100 万份文档中检索最相关的片段：

阶段1 (Bi-Encoder 召回): 100万 Doc → ANN 索引 → 选 Top-100 → ~5ms 阶段2 (Cross-Encoder 重排): 100个候选 → 每个 query-doc 对过 Cross-Encoder → 100 × 30ms = 3000ms ⚠️ 3000ms 不可接受！ 实际情况: 100个候选 → 先用 BM25+Dense 混合粗排(RRF) → 取 Top-20 → 20 × 30ms = 600ms → 仍在可接受范围内 或者: 用 Cohere Rerank API (支持 batch) → 100 个候选一次请求 → ~200ms
工程现实：Cross-Encoder 的延迟是线性的（除非服务端批量优化）。100 个候选在某些实现中就是 100 次前向传播。所以在混合检索阶段一定要把候选压缩到 20-30 个以内再送 Reranker。

Reranker 的实际效果

在金融文档 RAG 基准测试中（BGE-Reranker-v2-m3）：

策略Recall@10MRR延迟
纯 Dense (BGE-M3)52%0.34~5ms
混合 (Dense + BM25, RRF)78%0.51~10ms
混合 + Reranker Top-1086%0.68~350ms
混合 + Reranker Top-2091%0.74~650ms

Reranker 把 MRR 从 0.51 提升到 0.74——用户看到的相关文档排得更靠前，模型生成答案时"看到的"就是更高质量的信息。

4. GraphRAG：当向量检索不够"全局"

向量检索的边界

向量检索本质上是局部匹配——你问什么，它找语义最接近的片段。但如果问题是：

❌ "2025 年光伏行业可转债的整体转股率趋势如何？" → 没有一个"片段"能回答这个问题 ❌ "哪些转债的强赎条款和回售条款之间存在相互影响？" → 需要跨文档理解条款间的关系，不是片段匹配能解决的 ❌ "这只转债涉及的担保公司还担保了哪些其他的转债？" → 这是图遍历问题，向量检索只能找到含"担保"关键词的文档

这些都是 Global Query（全局查询）——需要跨文档的聚合、总结和关系推理。向量检索是 Local Query 的工具，不是 Global Query 的工具。

Microsoft GraphRAG 的流水线

GraphRAG 分两个阶段 (offline 和 online): Offline (索引构建): 文档 → Entity Extraction (LLM) → Relationship Extraction (LLM) → Knowledge Graph (Entity + Relation nodes) → Community Detection (Leiden algorithm) → Community Summary (LLM 为每个社区生成摘要) Online (查询): Local Query: 问题 → Embedding → 匹配实体 → 返回相关关系和社区摘要 Global Query: 问题 → 匹配社区摘要 → 返回跨文档的全局性答案

关键创新不是图本身——知识图谱在搜索领域早就有——而是 LLM 做实体/关系提取 和 社区摘要 这两步。传统 KG 需要人工设计 schema 和手工标注，成本极高。GraphRAG 用 LLM 自动发现实体和关系，不需要预定义 schema。

GraphRAG 的代价

向量 RAGGraphRAG
索引构建时间（1000 份文档）几分钟数小时（LLM 需要处理每一段文本做实体提取）
索引构建成本（API）$0.5$15-50（大量 LLM 调用）
查询延迟~100msLocal: ~200ms, Global: ~2s
适合的问题类型事实提取、条款查找总结、趋势、关系推理、全局模式
不适合多跳推理、全局统计精确关键词匹配（大材小用）
决策准则：90% 的可转债分析场景不需要 GraphRAG——混合检索 + Reranker 足够。只有当你的问题涉及"跨文档关系"（比如"找出所有受同一担保方影响的可转债"）时才值得上 GraphRAG。先用混合检索+Reranker 跑通，把 GraphRAG 作为 RAG 进阶的"高端武器"而不是"默认选项"。

5. 策略选择决策树

你的查询是什么类型？ │ ├─ 精确事实查找 ("下修触发条件是30天还是20天?") │ → 纯 BM25 可能就够了 │ ├─ 语义理解查找 ("这只转债的投资风险点在哪里?") │ → Dense + Reranker │ ├─ 混合："找条款"但表述多样 ("哪些条件会触发回售?") │ → 混合检索 (BM25 + Dense, RRF) + Reranker ← 最通用 │ ├─ 数值敏感查询 ("溢价率低于15%的可转债") │ → BM25 + 结构化过滤（在向量库 metadata 上做 filter，不要 │ 只在 prompt 里放，模型不擅长精确数值过滤） │ └─ 全局总结/关系推理 ("同业转债的条款差异趋势") → GraphRAG (Global Query)
