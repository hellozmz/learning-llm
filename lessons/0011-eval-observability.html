第11课：Eval + 可观测性

Agent 上线后，你怎么知道它在正常工作？模型不会"准确率 95%"这样直白地告诉你，你得自己造尺子。
预计时间：40 分钟  |  2026-06-29

1. 为什么传统 ML 的评估不够用

在传统 ML（包括你做的可转债 LightGBM 模型）里，评估是直接的：拿一个测试集，跑一遍，算 MAE / F1 / AUC。结果是一个数，比模型版本之间谁高谁低就行。

LLM 评估的难点是：输出是非结构化文本。"浦发转债当前溢价率 8.3%，建议持有"和"溢价率 8.3 个百分点，可继续持仓"——这两个答案都对，但你没法用 exact match 评判。更糟的是，模型有时会编造数字、漏掉关键步骤、引用错误的文档段落。

核心问题：LLM 评估需要回答三类问题——事实对不对（faithfulness）、回答切题不切题（relevance）、流程对不对（agent 工具选择）。每一类都需要不同的评估方法。

2. 离线评估：构造 Golden Set

离线评估的本质是：你预先准备一批 (输入, 期望输出) 对，每次模型或 prompt 改动后跑一遍，看输出和期望的吻合度。关键不在算法，在数据。

Golden Set 怎么构造

三个层次，从易到难： 1. 简单事实题 (50 条) Q: "浦发转债的到期日是？" A: "2027-03-31" ← 唯一正确答案，exact match 2. 推理题 (30 条) Q: "浦发转债当前溢价率 8%，正股 MA20 上行， 距到期 18 月，未触发强赎。按双低策略该怎么操作？" A: 关键点必须包含: "买入" + "溢价率低" + "正股趋势配合" ← 用 LLM-as-judge 评分 3. 边界题 (20 条) Q: "浦发转债溢价率 50%，正股已退市风险， 距到期 1 月，已公告强赎。" A: 关键点: "止损" + "高溢价风险" + "强赎避险" ← 测试模型在极端场景的判断

每条 golden 样本除了 (Q, A) 外，还要标注关键事实点——答案中必须出现的实体、数字、结论。这些点用来做自动评分。

3. LLM-as-Judge：用模型评模型

exact match 只能判简单事实题。对推理题和边界题，你需要一个能理解语义的评分器。最实用的方案是用一个更强的模型（Claude Opus / GPT-4）当评委。

JUDGE_PROMPT = """你是一个严格的评分员。下面是用户问题、参考答案、模型输出。
请按三个维度打分（0-2 分）：

1. 事实正确性 (factuality): 模型输出的事实是否和参考答案一致
   - 0: 有错误事实
   - 1: 事实正确但不完整
   - 2: 事实完全正确

2. 推理质量 (reasoning): 推理链是否合理
   - 0: 推理缺失或有逻辑错误
   - 1: 推理存在但跳跃
   - 2: 推理清晰且严密

3. 完整性 (completeness): 是否覆盖参考答案所有关键点
   - 0: 漏掉主要结论
   - 1: 漏掉次要细节
   - 2: 全部覆盖

输出 JSON: {"factuality": 0-2, "reasoning": 0-2, "completeness": 0-2, "issues": "..."}

用户问题: {question}
参考答案: {reference}
模型输出: {prediction}"""

def llm_judge(question, reference, prediction):
    resp = client.messages.create(
        model="claude-opus-4-6",
        system=JUDGE_PROMPT,
        messages=[{"role": "user", "content": f"Q: {question}\nRef: {reference}\nPred: {prediction}"}],
    )
    return json.loads(resp.content[0].text)
LLM-as-judge 的三个坑：
1. 位置偏置：评委倾向于给"第一个出现的答案"更高分。解法：随机打乱顺序跑两次取平均。
2. 长度偏置：评委倾向于给更长的输出更高分（看起来更"用心"）。解法：在 prompt 里明确"长度不影响评分"，并限制输出字数。
3. 自我偏好：用 Claude 评 Claude 输出会偏高。解法：跨模型评（用 GPT-4 评 Claude，反之亦然）。

4. RAG 专项评估：Ragas 框架

对 RAG 系统的评估有标准化框架，最常用的是 Ragas。它把 RAG 拆成四个维度，每个维度有专门的评估方法：

维度测什么评估方法
Faithfulness（忠实度） 答案是否基于检索到的文档，没有编造 把答案拆成句子，每句验证能否在 context 中找到支撑
Answer Relevance 答案是否回应了用户问题 从答案反推可能的问题，和原问题计算相似度
Context Precision 检索到的文档里有多少是真正相关的 对每个 retrieved chunk 标注是否相关，算 precision
Context Recall 答案需要的信息有多少被检索到了 把参考答案拆成事实点，验证每个点是否在 context 中
Faithfulness 评估的具体做法： 答案: "浦发转债溢价率 8.3%，到期日 2027-03-31" ↓ 拆成原子陈述 陈述1: "浦发转债溢价率 8.3%" 陈述2: "浦发转债到期日 2027-03-31" ↓ 对每个陈述，让 LLM 验证 context 中是否有支撑 Context: "...浦发转债当前价格 105.2，溢价率 8.3%..." → 陈述1 有支撑 ✓ → 陈述2 无支撑 ✗ (context 里没提到期日) ↓ Faithfulness = 1/2 = 0.5 关键: 忠实度低 = 模型在编造，这是 RAG 最致命的失败模式

5. Agent 专项评估：工具选择准确率

Agent 的评估比 RAG 更复杂，因为多了"决策"这一层。你需要评估的不只是最终答案，还有中间过程。

三个层次的 Agent 评估

层次1: 工具选择准确率 (Tool Selection Accuracy) - 给定一个问题，模型选对了哪个工具？ - 需要 golden trace: "这个问题应该先调 A 再调 B" - 准确率 = 选对的步骤数 / 总步骤数 层次2: 参数正确率 (Parameter Accuracy) - 选对工具后，参数对不对？ - "浦发转债" vs "浦发银行转债" vs "浦发转债(SH110062)" - 用模糊匹配或 exact match 层次3: 端到端任务完成率 (Task Success) - 最终答案是否解决了用户问题？ - 二元判定（成功/失败）或 1-5 分打分 
def evaluate_agent_trace(trace, golden_trace):
    """trace: 实际运行记录, golden_trace: 预期路径"""
    tool_acc = 0
    param_acc = 0
    n = min(len(trace), len(golden_trace))

    for i in range(n):
        actual = trace[i]
        expected = golden_trace[i]

        if actual.tool_name == expected.tool_name:
            tool_acc += 1
            # 工具选对后再看参数
            if param_match(actual.args, expected.args):
                param_acc += 1

    return {
        "tool_accuracy": tool_acc / len(golden_trace),
        "param_accuracy": param_acc / max(tool_acc, 1),
        "step_count_match": len(trace) == len(golden_trace),
    }
Agent 评估的难点：同一个问题可能有不同的合法路径——比如先查价格再查评级，或先查评级再查价格，都对。Golden trace 不能只存一条路径，要存一个"合法路径集合"。这比 RAG 评估要复杂得多。

6. 在线可观测性：Trace 和 Metrics

离线评估告诉你"模型能力有没有变化"，在线可观测性告诉你"现在线上发生了什么"。两者不可替代。

Trace：每次调用的完整链路

一次 Agent 调用的 trace： [trace_id: abc123] ├─ LLM call #1 (3.2s, 850 tokens) │ └─ tool_call: query_cb_price("浦发转债") ├─ Tool execution (0.4s) │ └─ result: {"price": 105.2, "premium": 8.3} ├─ LLM call #2 (2.8s, 720 tokens) │ └─ tool_call: query_stock_trend("600000") ├─ Tool execution (0.6s) │ └─ result: {"trend": "upward"} └─ LLM call #3 (1.5s, 420 tokens) └─ final_answer: "建议持有..." 总耗时: 8.5s, 总 tokens: 1990, 工具成功率: 2/2

每个 trace 节点记录：开始/结束时间、token 数、成本、输入输出、错误。当用户投诉"AI 给的答案不对"时，trace 是你定位问题的唯一线索。

四个核心指标

指标含义异常说明什么
Token Usage 每次调用的 input/output token 数 突增 = prompt 变长或上下文堆积
Latency (P50/P95/P99) 首 token 延迟 (TTFT)、总时长 P99 飙升 = 模型超时或重试
Tool Success Rate 工具调用成功 / 总调用 下降 = 工具端 bug 或参数错误
Error Rate 按错误类型分类（JSON 解析失败、超时、API 限流） 某类激增 = 具体故障源

7. 工具选型：Langfuse vs Phoenix vs OpenTelemetry

2026 年主流的 LLM 可观测性工具：

工具特点适用场景
Langfuse 开源、自部署、LLM 应用专用 个人/小团队，数据不出内网
Phoenix (Arize) 开源 + 商业版，强项是 eval 和 drift 检测 需要离线 eval 和在线监控一体化
OpenTelemetry 通用可观测性标准，LLM 之上需自己扩展 已有完整 OTel 体系，不想引入新工具
LangSmith (LangChain) 商业 SaaS，和 LangChain 深度集成 用 LangChain 框架且不介意托管
可转债项目选型建议：用 Langfuse 自部署。理由：数据敏感（转债持仓和策略不能外泄）、开源免费、对 Agent trace 原生支持、Python SDK 接入只需 3 行代码。LangSmith 商业版会让数据出内网，不符合你的项目约束。

8. 实战：给可转债 Agent 接入 Langfuse

# 第一步：装包
# pip install langfuse anthropic

from langfuse import Langfuse
from langfuse.decorators import observe
from anthropic import Anthropic

langfuse = Langfuse()  # 从环境变量读 host/public_key/secret_key
client = Anthropic()

@observe()  # 这个 decorator 自动记录输入输出、token、耗时
def cb_agent(question: str) -> str:
    messages = [{"role": "user", "content": question}]
    tools = load_cb_tools()

    for step in range(10):
        resp = client.messages.create(
            model="claude-opus-4-6",
            messages=messages,
            tools=tools,
        )

        if resp.stop_reason == "end_turn":
            return resp.content[0].text

        # 把工具调用也作为子 trace 记录
        for tool_use in resp.content:
            if tool_use.type == "tool_use":
                result = dispatch_tool_with_observation(tool_use)  # 内部也用 @observe
                messages.append({"role": "assistant", "content": resp.content})
                messages.append({"role": "user", "content": [{
                    "type": "tool_result",
                    "tool_use_id": tool_use.id,
                    "content": str(result),
                }]})

    return "未完成"

# 运行后，去 Langfuse UI 看完整 trace
cb_agent("浦发转债现在值得买吗？")

接入后，每次 Agent 调用在 Langfuse UI 上能看到完整链路：哪一步调了什么工具、每步耗时和 token、最终答案。出问题时直接定位到具体步骤。

9. 回归测试：CI 流水线里的 Eval

把离线评估做成 CI 步骤，每次改 prompt 或换模型版本时自动跑：

# .github/workflows/llm-eval.yml 或 scripts/run_eval.py

def regression_test():
    golden_set = load_golden_set("tests/golden_set.json")
    results = []

    for sample in golden_set:
        pred = cb_agent(sample.question)
        scores = llm_judge(sample.question, sample.reference, pred)
        results.append({
            "id": sample.id,
            "scores": scores,
            "passed": all(v >= 1 for v in scores.values() if isinstance(v, int))
        })

    pass_rate = sum(r["passed"] for r in results) / len(results)

    # 关键：和上一版本的 baseline 比较
    baseline = load_baseline("tests/baseline.json")
    if pass_rate < baseline - 0.05:
        raise Exception(f"回归: {pass_rate:.2%} < baseline {baseline:.2%}")

    save_baseline(pass_rate)
    print(f"通过率 {pass_rate:.2%}")

if __name__ == "__main__":
    regression_test()
Baseline 比较是关键：不要追求 100% 通过率——LLM 总会有边界 case 失败。要做的是和上一次发布的 baseline 比，不允许退化超过 5%。每次模型升级（claude-opus-4 → claude-opus-5）或 prompt 改动都跑一遍，确认没有回归。

10. 自检

问题 1/4 — 评估方法选择

你的可转债 RAG 系统上线后，用户反馈"AI 给的到期日和募集说明书对不上"。要定位这个问题，最直接的评估指标是？

A — Answer Relevance，看答案是否切题
B — Context Precision，看检索是否精确
C — Faithfulness，看答案是否忠于文档
D — Tool Success Rate，看工具调用成功率

问题 2/4 — LLM-as-judge 偏置

用 Claude 当评委评估 GPT-4 输出和 Claude 输出，发现 Claude 给 Claude 的平均分高 0.4。这种现象最准确的描述是？

A — 位置偏置，需要打乱顺序重跑
B — 长度偏置，需要限制输出字数
C — 自我偏好，需要跨模型交叉评估
D — 评分标准模糊，需要细化 rubric

问题 3/4 — Agent 评估难点

评估 Agent 时，你发现同一个问题"分析浦发转债"有两条合法路径：先查价格再查评级、或先查评级再查价格。直接用单条 golden trace 算 tool accuracy 会怎样？

A — 高估准确率，因为合法路径都被算对
B — 低估准确率，合法的替代路径被算错
C — 不影响准确率，只影响步骤数匹配
D — 仅影响参数评分，不影响工具选择

问题 4/4 — 在线监控设计

你的 Agent 上线后，发现 P95 延迟从 5s 涨到 12s，但 P50 没变。Token Usage 和 Tool Success Rate 都正常。最可能的故障源是？

A — 模型变慢，所有请求都受影响
B — 工具端故障导致重试，影响所有调用
C — Prompt 变长导致每次调用 token 增加
D — 少数复杂请求多走几步循环拉高尾部延迟
